## 项目愿景

**特许鹰眼（Franchise Sentinel）** 是一个面向特许经营律师的自动化法律风险情报监控系统。

**核心价值主张：**
在客户发现特许经营法律风险之前，系统已经自动发现、筛选、评估并推送到律师的企业微信群，让律师能够第一时间为客户提供专业建议。

**长期目标：**
1. **全面覆盖**：覆盖中国特许经营领域的主要信息源（政府公告、媒体报道、社交媒体、司法文书）。
2. **精准推送**：误报率 <10%，漏报率 <5%。
3. **零运维**：部署后无需人工干预，自动运行、自动修复、自动升级。
4. **可扩展**：支持新增品牌监控、新增数据源、新增推送渠道。

## 架构决策记录（ADR）

### ADR-001：搜索优先于直接爬取

**决策：** 通过百度新闻 + 搜狗微信搜索来发现内容，而不是直接爬取目标网站。

**原因：**
- 百度/搜狗已经做了反爬对抗和内容索引，我们直接复用它们的搜索结果更稳定。
- 直接爬取网易/新浪/腾讯等网站维护成本高、容易被封。
- 搜索词可以通过 `site:` 过滤指定来源，实现"间接定向抓取"。

**代价：**
- 搜索结果有延迟（通常 15 分钟到数小时）。
- 百度搜索结果受 SEO 影响，可能遗漏。
- 搜狗微信搜索对微信内容覆盖较好，但非微信内容较弱。

**未来：** 当搜索覆盖不足时，可以增加少量高价值直连源（如市监总局、CCFA）。

### ADR-002：评分制过滤而非关键词白名单

**决策：** 使用加权评分系统（`score_article`）而非简单的关键词包含/排除。

**原因：**
- 关键词白名单容易误杀（如"加盟"一词在餐饮新闻中大量出现，但不一定是法律风险）。
- 评分制可以综合多维度信号：关键词权重、品牌命中、来源权威度、风险标签。
- 分数阈值可调，方便根据推送质量动态调整。

**当前阈值：**
- 最低推送分数：6 分
- 品牌命中 + 风险标签：即使 <6 分也强制推送

**未来：** 引入机器学习模型替代规则评分，但前提是积累足够多的标注数据。

### ADR-003：时效性优先于全面性

**决策：** 宁可漏掉一些旧新闻，也绝不推送过期内容。

**原因：**
- 用户反馈：2020 年、2023 年的新闻推送严重降低工具可信度。
- 特许经营法律风险具有时效性：行政处罚决定书、法院判决、备案撤销等事件，旧闻对当前客户决策帮助有限。

**当前策略：**
- 默认 30 天窗口（`max_article_age_days: 30`）。
- 无法识别时间的文章直接跳过（`require_publish_date: true`）。
- 正文不参与时间判断，避免页脚推荐文章干扰。

**未来：** 可以针对不同类型的风险设置不同窗口：行政处罚 30 天、法院判决 90 天、行业政策 180 天。

### ADR-004：云端状态仓库保证跨次持久化

**决策：** 使用独立的 GitHub 私有仓库 `franchise-sentinel-state` 保存 `pushed_history.json` 和 `daily_findings.json`。

**原因：**
- GitHub Actions 每次运行在全新沙箱中，本地文件不持久。
- 去重历史和每日简报线索必须跨次保留。
- GitHub 私有仓库是零成本、零运维的持久化方案。

**替代方案：**
- GitHub Gist：需要 `gist` 权限，用户令牌没有。
- 对象存储（OSS/S3）：需要额外账号和费用。
- 本地数据库：GitHub Actions 沙箱不支持。

### ADR-005：GitHub Actions 作为主力自动化引擎

**决策：** 使用 GitHub Actions 而非 TRAE 自动化作为定时任务引擎。

**原因：**
- GitHub Actions 专为 CI/CD 和定时任务设计，稳定性更好。
- 日志完整、可追溯、可调试。
- Secrets 管理更安全（企微 webhook 不暴露）。
- 与代码仓库天然集成，workflow 文件版本可控。

**TRAE 自动化角色：** 作为备用或开发调试工具，不用于生产定时任务。

### ADR-006：企微 Webhook 作为唯一推送渠道

**决策：** 当前只支持企业微信机器人推送。

**原因：**
- 用户（律师）的日常沟通工具就是企微。
- Webhook 推送零成本、零延迟。
- 企微支持 Markdown 格式，信息展示效果好。

**未来：** 可以增加邮件、短信、飞书等渠道，但企微始终是主渠道。

### ADR-007：Python 标准库优先

**决策：** 尽量使用 Python 标准库，减少外部依赖。

**原因：**
- 减少依赖安装时间和兼容性问题。
- GitHub Actions 每次运行都是全新环境，依赖越少越快。

**当前依赖：** `requests`, `beautifulsoup4`, `lxml`。`schedule` 仅在本地手动运行时需要。

## 扩展路线图

### Phase 1：稳定运行（当前）
- 61 条搜索词覆盖主流平台
- 380 个重点品牌监控
- 30 天时效过滤
- 企微推送
- GitHub Actions 自动化

### Phase 2：质量提升
- [ ] 增加自动化测试套件（`test_franchise_monitor.py`）
- [ ] 引入机器学习评分模型（需要标注数据）
- [ ] 增加更多直连源（如最高法公众号、各高院公众号）
- [ ] 支持多推送渠道（邮件、飞书）

### Phase 3：智能化
- [ ] 自动发现新品牌（从新抓取文章中提取品牌名，与 CCFA 列表比对）
- [ ] 风险事件关联分析（同一品牌的多条线索自动聚类）
- [ ] 生成结构化报告（PDF/Word 格式，可直接发给客户）
- [ ] 支持用户自定义监控规则（通过 Web 界面或配置文件）

### Phase 4：平台化
- [ ] 多用户支持（不同律师团队各自配置）
- [ ] 数据看板（历史推送统计、品牌风险趋势）
- [ ] API 接口（供第三方系统调用）

## 质量门（Quality Gates）

任何改动必须通过以下检查：

1. **语法检查**：`python3 -m py_compile franchise_monitor.py`
2. **干跑测试**：`python3 franchise_monitor.py --once --dry-run` 不报错
3. **状态同步测试**：`python3 state_sync.py pull` 和 `push` 正常
4. **GitHub Actions 验证**：workflow 文件推送到 GitHub 后能手动触发成功
5. **企微推送验证**：实际运行后企微收到推送
